# 店掌柜阶段成果汇报

## 项目目标

「店掌柜」智慧零售管理系统面向乡镇小店和中老年经营者，目标不是做复杂 ERP，而是先完成简单、清楚、少步骤的经营工具。

当前阶段重点是打通小店日常经营中最常见的链路：

```text
商品建档 -> 采购入库 -> 收银销售 -> 库存扣减 -> 赊账还款 -> 看板查看
```

## 当前成果

### 1. 完成店铺经营模块

系统已新增独立的店铺经营模块，包含：

- 首页看板
- 收银台
- 商品管理
- 库存管理
- 客户管理
- 赊账管理
- 销售记录
- 采购入库
- 供应商管理
- 经营报表
- 提醒中心

这些功能覆盖了小店老板每天最常用的经营动作。

### 2. 完成模块化拆分

后端已从单一大控制器拆分为按业务命名的 Controller 和 Service。前端也从一个 API 文件拆成商品、库存、客户、收银、赊账、采购、供应商、报表、提醒等多个文件。

这样做的好处是后续二次开发时可以按模块定位问题，不需要在一个大文件里反复查找。

### 3. 完成商品分类问题修复

商品分类下拉无数据的问题已经通过数据补齐和前端提示处理：

- 数据库补入常用分类
- 前端商品页加载分类下拉
- 无分类时提示先新增分类
- 商品列表显示分类名称

### 4. 完成商品与供应商绑定

系统支持一个商品绑定多个供应商。这样更贴近小店实际情况：同一种商品可能从不同批发商进货，采购时可以按供应商优先筛选商品。

当前关系为：

- 商品属于一个分类
- 商品可以绑定多个供应商
- 分类和供应商不直接绑定

### 5. 完成商品图片上传接入

商品图片上传复用项目已有 OSS 能力，不把外部阿里云工具类直接放入代码。

当前方案：

- 商品表保存 OSS 文件 ID
- 前端使用统一图片上传组件
- 上传走资源服务
- 阿里云密钥只放运行环境或系统配置，不进入 Git

### 6. 完成适老化界面初步优化

商品、供应商、采购等页面已按适老化方向调整：

- 字号更大
- 按钮更高
- 标签更清楚
- 表单字段减少
- 操作完成有明确提示
- 常用入口前置

设计重点是让经营者“看得懂、点得准、少走弯路”。

## 难点说明

### 1. 不是只做页面

这个项目的页面入口来自数据库菜单。只写前端页面还不够，还要同步：

- 后端接口
- 权限标识
- 数据库菜单
- 角色授权
- 前端路由组件

所以每新增一个业务页面，都需要前后端和 SQL 一起考虑。

### 2. 数据库要分清职责

系统基础数据和菜单仍在 `dzg_cloud`，店铺业务数据放在 `dzg_shop`。如果把业务表和系统表混在一起，后续维护会越来越困难。

当前处理方式是：

- `dzg_cloud`：用户、菜单、权限、系统配置
- `dzg_shop`：商品、库存、客户、订单、赊账、采购、供应商

### 3. 图片上传不能直接写密钥

外部 `AliOssUtil.java` 虽然能快速上传，但里面可能包含明文密钥。直接并入仓库会有安全风险，也容易被 GitHub secret scanning 拦截。

最终选择复用框架已有 OSS 能力，这是更安全、也更符合项目结构的做法。

### 4. 适老化不只是放大字号

适老化设计还包括：

- 常用功能放在更容易点的位置
- 每页只突出一个主操作
- 表单只保留必要字段
- 金额、库存、欠款等关键数字更醒目
- 错误提示要直接说明怎么处理

## 有趣的地方

### 首页看板的取舍

一开始系统里可能出现两个看板：顶部首页看板和店铺经营里的首页看板。为了减少用户困惑，当前保留顶部首页看板作为主要入口，隐藏店铺经营里的重复看板。

这个取舍更适合小店老板：进系统后先看到今天营业额、订单数、赊账和库存预警，再通过大按钮进入收银、入库、还款等常用功能。

### 供应商和商品做成多对多

如果一个商品只能绑定一个供应商，会限制真实进货场景。小店经常会因为价格、库存、距离等原因从不同供应商进同一种商品。

因此当前采用商品-供应商关联表，让采购入库可以按供应商筛商品，但不限制老板临时选择其他商品。

### 库存服务成为核心

收银会扣库存，采购会加库存，盘点会调库存。看起来是不同页面，本质上都在改库存。

所以库存变化集中在库存服务处理，并写入库存流水。后续做退货、报损、盘点时可以继续复用这条主线。

## Git 提交策略

后续不再把多个模块集中到一次大提交里。推荐按下面方式小步提交：

```text
docs: 整理店掌柜当前实现文档
fix: 修复商品分类和供应商联动问题
feat: 完善商品图片上传和展示
feat: 优化收银和采购入库流程
feat: 完善经营报表和提醒中心
```

每次提交前检查：

- 不提交真实密码
- 不提交 `target`、`dist`、`node_modules`
- 前端改动跑构建
- 后端改动跑店铺模块编译
- SQL 改动写补丁并同步数据库
